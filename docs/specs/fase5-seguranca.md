# Fase 5 — Segurança e Privacidade (Security & Privacy by Design)
## Produto: FireAudit — v1 (produto completo, inclui escrita remota — ver nota abaixo)

Este documento formaliza os pontos já sinalizados nas fases anteriores (Fase 1, item 2.1; Fase 3, seção 7; Fase 4, seção 7) e cobre o restante da superfície de segurança necessária para um produto que audita firewalls de terceiros. Princípio geral: FireAudit é, por definição, um alvo de alto valor (Fase 1, item 2.1) — a postura de segurança não é um checklist de compliance, é a própria proposta de valor do produto (um SaaS de segurança que é ele mesmo inseguro não sobrevive à primeira violação divulgada).

**Atualização de 2026-07-08 — escopo do v1 passou a incluir escrita remota de regras (tier Premium, antes previsto só para v5):** isso muda esta fase de forma material, não cosmética — um produto que só lê configuração de firewall e um produto que também escreve configuração de firewall têm modelos de ameaça fundamentalmente diferentes (o segundo tem uma capacidade real de dano ao cliente, não só de vazamento de informação). As seções 1, 2 e a nova seção 11 foram reescritas para tratar a escrita remota como parte do núcleo de segurança do produto desde o lançamento, não como um adendo.

## 1. Modelo de ameaças (resumo orientado a decisão)

| Ativo | Ameaça principal | Impacto se comprometido |
|---|---|---|
| `agent_token` de ingestão (por firewall) | Vazamento via log, interceptação em trânsito, ou comprometimento do backend | Atacante pode enviar snapshots falsos para aquele firewall específico — gera achados/alertas falsos. **Não** dá acesso ao firewall real, e — ponto crítico desde esta atualização — **também não dá capacidade de enviar comandos de escrita**, porque o token de ingestão e o de comando de escrita são escopados separadamente (seção 2). Escopo do dano é limitado por desenho. |
| Payload do snapshot (regras, topologia) | Interceptação em trânsito, acesso não autorizado ao banco | Vazamento de informação de configuração de rede de clientes — grave para reputação, mas não concede controle sobre o firewall |
| **Fila de comandos remotos (`firewall_commands`)** | Comprometimento do backend, sessão de usuário roubada, ou erro de autorização que permita criar/confirmar um comando para o firewall errado | **Este é o ativo de maior impacto potencial do produto desde 2026-07-08** — diferente dos outros itens desta tabela, aqui o dano não é vazamento de informação, é alteração real da postura de segurança do firewall do cliente (ex: criar uma regra `any-any`, apagar uma regra de bloqueio). Tratado como cenário-mestre a mitigar junto com o backend/infra (linha abaixo) — ver seção 11 para os controles específicos. |
| Credenciais de usuário (login do FireAudit) | Credential stuffing, phishing, senha fraca | Acesso à conta do cliente no dashboard — visualização de dados sensíveis (Pro+) e, desde esta atualização, **capacidade de criar e confirmar comandos de escrita remota** se a conta tiver tier Premium — eleva o impacto de uma conta comprometida de "vazamento de dados" para "alteração de firewall de terceiro", motivo pelo qual a seção 11 trata 2FA como obrigatório para esse fluxo específico. |
| Segredos de webhook (`alert_channels.config`) | Vazamento do banco (dump, backup mal protegido) | Atacante poderia enviar mensagens falsas para o Slack/Discord do cliente se o segredo de assinatura vazar |
| Backend/infraestrutura (VPS, banco) | Exploração de vulnerabilidade na aplicação, credenciais de infra vazadas | Pior cenário: acesso a todos os `agent_tokens` e payloads de todos os clientes simultaneamente, **e** capacidade de injetar comandos de escrita na fila de qualquer firewall — por isso é tratado como o cenário-mestre a mitigar, agora com peso ainda maior do que antes desta atualização. |

**Mudança de modelo de ameaças em relação à versão anterior deste documento:** a versão anterior afirmava que "comprometimento do firewall do cliente via FireAudit" não estava no modelo de ameaças porque o produto era read-only. Isso não é mais verdade — a escrita remota agora é parte do v1 (Fase 2, nota de escopo), então esse vetor existe desde o lançamento e é tratado com o rigor de um projeto de segurança de primeira classe, não como um adendo posterior (ver seção 11).

## 2. Autenticação e autorização

- **Usuários (dashboard):** email/senha com hash `argon2id` (preferível a bcrypt por resistência a ataques com GPU/ASIC; custo computacional aceitável para o volume esperado). Sessão via JWT de curta duração (15 min) + refresh token (rotação a cada uso, revogável). Rate limiting específico no endpoint de login (ver seção 5) para mitigar credential stuffing.
- **2FA — mudou de "adiado para v2" para "obrigatório no v1, mas só para o fluxo de escrita remota" (2026-07-08):** a razão da mudança está na seção 1 — uma conta comprometida sem 2FA e com tier Premium agora significa "atacante pode alterar o firewall do cliente", não só "atacante vê dados". Desenho final: 2FA (TOTP, ex: Google Authenticator) é **opcional para login geral** (mantém a fricção de onboarding baixa, Fase 1 seção 5, para quem só usa leitura/dashboard), mas **obrigatório e reforçado no momento de confirmar qualquer `firewall_command`** (seção 11) — ou seja, mesmo um usuário que fez login sem 2FA é obrigado a completar o desafio de 2FA especificamente no passo de confirmação de uma escrita remota. Isso equivale a um "step-up authentication", padrão comum em produtos financeiros para a mesma classe de problema (ação de alto impacto vs. navegação comum). Campos `two_factor_secret`/`two_factor_enabled` (Fase 4) continuam os mesmos; o que muda é que `two_factor_enabled = true` passa a ser pré-requisito de aplicação para o usuário conseguir usar qualquer funcionalidade de escrita, não um recurso opcional.
- **Agente (ingestão):** autenticado por `agent_token` no header (`Authorization: Bearer <token>`), validado contra `agent_tokens.token_hash`. Sem sessão, sem cookie — cada requisição é autenticada isoladamente (stateless), coerente com a natureza de job assíncrono do agente.
- **Agente (polling de comandos remotos) — novo (2026-07-08):** usa o **mesmo `agent_token`** de ingestão (não um segundo token) para chamar `GET /v1/firewalls/{id}/commands/pending`, mas o **escopo de ação de cada credencial não é o mesmo**: o `agent_token` autentica "este agente pode ler/escrever no contexto deste firewall_id", enquanto a autorização de *qual comando* pode ser entregue é decidida inteiramente no backend, no momento em que o comando foi criado — o agente nunca decide o que fazer, só executa o que já foi confirmado por um usuário humano (seção 11) e devolvido pelo polling. Isso significa que possuir o `agent_token` vazado não é suficiente para *criar* um comando de escrita — só permite *ler* (fazer polling de) comandos que um usuário autenticado já confirmou via dashboard. A criação/confirmação do comando exige sessão de usuário (JWT) com 2FA, nunca o `agent_token`. Essa separação é o controle mais importante desta seção — repetido explicitamente porque é fácil de implementar errado (ex: um desenvolvedor apressado poderia reusar o mesmo middleware de autenticação do endpoint de ingestão para o endpoint de criação de comando, o que romperia essa separação).
- **Autorização:** verificação de "o recurso pertence à `organization_id` do usuário autenticado" em toda query — centralizado na camada de `application` (casos de uso), nunca deixado para cada endpoint decidir individualmente. Isso evita o erro mais comum de multi-tenant mal feito: um endpoint que esquece o filtro de organização e permite acesso cross-tenant. Para o caso específico de `firewall_commands`, a mesma centralização se aplica com um requisito adicional: o caso de uso de criação/confirmação de comando deve verificar tier (Premium) **e** 2FA habilitado **e** propriedade do firewall, os três no mesmo ponto único — não em três verificações espalhadas que podem divergir com o tempo. Quando RBAC (v2) chegar, esse ponto único de verificação é o lugar certo para adicionar granularidade de papel, sem reescrever toda a API.

## 3. Proteção de dados em repouso

- **`agent_tokens.token_hash` e `users.password_hash`:** hash, nunca reversível — já coberto na Fase 4.
- **`alert_channels.config` (segredos de webhook):** criptografado em repouso via envelope encryption com chave gerenciada por KMS (ex: OCI Vault, já que a hospedagem é OCI — Fase 3). O backend nunca guarda a chave de criptografia junto com o dado; a chave fica no KMS, o banco guarda só o texto cifrado. Justificativa de que isso *não* é overengineering: diferente de uma senha (que pode ser só hash, nunca precisa ser lida de volta), o valor de um webhook precisa ser recuperado em texto claro para ser usado no envio do alerta — por isso precisa ser criptografia reversível, não hash, e reversível com chave gerenciada separadamente é a única forma correta de fazer isso.
- **`raw_payload` (snapshots):** não contém segredos (a API key do pfSense nunca chega até aqui, por desenho da Fase 4), mas contém topologia de rede — tratado como dado confidencial de cliente, protegido pelo controle de acesso de autorização (seção 2), não por criptografia adicional de coluna (seria redundante com a criptografia de disco do provedor de nuvem, que já cobre o "backup roubado" — ver seção 8).
- **Backups:** criptografados em repouso pelo provedor (OCI) — suficiente para o volume/estágio atual; não introduzir criptografia de aplicação adicional sobre backups agora (custo operacional sem ameaça correspondente no modelo desta fase).

## 4. Proteção de dados em trânsito

- TLS 1.2+ obrigatório em toda comunicação agente↔backend e navegador↔backend (já mencionado na Fase 2, RNF01) — reforçando aqui: **HSTS habilitado**, certificados via Let's Encrypt com renovação automática (custo zero, coerente com orçamento de USD 20/mês).
- **Verificação de integridade do payload do agente** (proposta na Fase 1, seção 3): o agente calcula um HMAC-SHA256 do payload usando uma chave derivada do `agent_token`, enviado em um header separado (`X-Signature`). O backend recalcula e compara antes de aceitar o snapshot. Isso protege contra dois cenários distintos do TLS: (a) um proxy/MITM que conseguiu interceptar mas não decifrar ainda tentar reenviar payload adulterado, e (b) mais realisticamente, detectar corrupção de dados em trânsito por bug/rede instável antes que ela gere um "achado" falso no motor de análise — o segundo caso é o motivo prático mais provável de isso disparar no dia a dia, não o primeiro.

## 5. Rate limiting e proteção de abuso

| Endpoint | Limite sugerido | Motivo |
|---|---|---|
| `POST /auth/login` | 5 tentativas / 15 min por IP+email | mitigar credential stuffing/brute force |
| `POST /ingest/snapshot` | 1 por minuto por `agent_token` (folga sobre o intervalo de Cron esperado) | evitar que um agente com bug em loop sobrecarregue o worker de fila |
| Endpoints de API pública (dashboard) | limite geral por usuário autenticado (ex: 100 req/min) | proteção padrão contra scraping/abuso, não é uma ameaça específica identificada, é higiene básica |

Implementação recomendada: no nível da aplicação (biblioteca simples, ex: `slowapi` para FastAPI) no MVP — **não** introduzir um API Gateway dedicado (Kong, etc.) agora; seria a mesma lógica de overengineering já rejeitada para microsserviços na Fase 3. Reconsiderar se/quando múltiplas instâncias do backend exigirem rate limit compartilhado entre processos (nesse ponto, mover o contador para Redis já resolve, sem gateway dedicado).

## 6. Validação de entrada e superfície de ataque da API

- Todo payload de entrada (login, cadastro de firewall, config de webhook, e o payload do agente) validado via schema Pydantic (nativo do FastAPI, Fase 3) — rejeição automática de campos inesperados/malformados antes de chegar à camada de aplicação.
- `raw_payload` do agente é JSON estruturado, não HTML/texto livre — risco de XSS armazenado é mitigado na origem, mas o frontend (Next.js/React) ainda deve tratar qualquer campo de texto livre exibido (ex: nome de regra de firewall definido pelo cliente) como não confiável na renderização (React já escapa por padrão; o cuidado real é não usar `dangerouslySetInnerHTML` em nenhum lugar que exiba dado de snapshot).
- SQL injection: mitigado por padrão pelo uso de ORM/query builder parametrizado (ex: SQLAlchemy) — nenhuma query com concatenação de string de input do usuário, ponto coberto pela própria estrutura do Repository Pattern (Fase 3).
- CORS: restrito ao domínio do frontend oficial (`app.fireaudit.io` ou equivalente) — sem wildcard.
